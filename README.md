# Система распределенного сбора и мониторинга журналов событий Windows

## Введение
Данный программный комплекс представляет собой клиент-серверное решение, предназначенное для автоматизированного сбора, агрегации и визуализации критических событий операционных систем семейства Windows. Система ориентирована на повышение информационной безопасности и оперативного выявления системных сбоев в локальной сети предприятия.

## Архитектура системы
Комплекс состоит из трех ключевых компонентов:
1. **Клиентская часть (Windows Service):** Фоновая служба, разработанная на платформе C# (.NET 10). Производит непрерывный мониторинг системных журналов (Application, System, Security), собирает показатели производительности системы (CPU, RAM, состояние дисков) и транслирует эти данные в форме параметризованных запросов на сервер. Обладает встроенным механизмом локального кэширования для предотвращения потери данных при разрыве сетевого соединения.
2. **Серверная часть (Ubuntu Server):** Веб-приложение на базе ASP.NET Core (.NET 10), функционирующее в качестве службы `systemd` на ОС Ubuntu 24.04. Принимает транзакции от клиентов на выделенном порту 8080 и предоставляет веб-интерфейс для оператора на порту 80.
3. **База данных (MariaDB):** Реляционная СУБД, обеспечивающая структурированное хранение записей системных журналов, метрик производительности и данных аудита безопасности.

## Функциональные возможности
* Мониторинг системных журналов Windows (Application, System) в реальном времени.
* Аудит событий информационной безопасности (отслеживание недокументированных попыток входа в систему, Event ID 4625).
* Сбор базовых метрик производительности узлов (загрузка процессора, остаток оперативной памяти, свободное место на дисках).
* Локальное кэширование транзакций (Offline Cache) на стороне клиента с последующей синхронизацией после восстановления связи.
* Веб-панель мониторинга с возможностью сортировки записей по временной шкале и фильтрации по источникам и типам журналов.

## Порядок развертывания

### Серверный компонент (Ubuntu 24.04)
1. Выполнить импорт структуры базы данных с помощью скрипта инициализации:
   ```bash
   mariadb -u root -p < init.sql
   ```
2. Разместить исполняемые файлы сервера в директории `/var/www/server_listener` и настроить права доступа для системного пользователя `www-data`:
   ```bash
   sudo chown -R www-data:www-data /var/www/server_listener
   ```
3. Зарегистрировать и запустить системную службу `server-listener.service`:
   ```bash
   sudo systemctl daemon-reload
   sudo systemctl enable server-listener
   sudo systemctl start server-listener
   ```

### Клиентский компонент (Windows)
1. Сконфигурировать адрес сервера и отслеживаемые параметры в файле `appsettings.json`.
2. Зарегистрировать исполняемый файл `error_catcher.exe` в качестве системной службы Windows, запустив PowerShell от имени Администратора:
   ```powershell
   New-Service -Name "ErrorCatcher" -BinaryPathName "C:\path\to\error_catcher.exe" -StartupType Automatic -DisplayName "Windows Error Catcher Service"
   ```
3. С целью обеспечения стабильности сетевого соединения на ранних этапах загрузки ОС, установить зависимость службы от сетевого стека:
   ```cmd
   sc.exe config "ErrorCatcher" depend= Nla
   ```
4. Запустить службу:
   ```powershell
   Start-Service -Name "ErrorCatcher"
   ```
